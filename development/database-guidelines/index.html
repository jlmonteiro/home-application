    Database Guidelines

    Overview

    This document defines the standards for the PostgreSQL 17 database, including schema design, naming conventions, migration rules, and data integrity patterns. All schema changes are managed by Liquibase — Hibernate is configured with ddl-auto: validate and never modifies the schema.

    1. Schema Architecture

    The database is organized into two logical schemas that enforce domain boundaries:

    Schema Purpose Examples
    profiles Identity, authentication, family structure user, user_profile, family_roles, age_group_config
    shopping Collaborative shopping experience shopping_lists, shopping_items, shopping_stores, coupons

    Cross-Schema References

    Cross-schema foreign keys are allowed for domain relationships (e.g., shopping.shopping_lists.created_by → profiles.user.id). Keep these minimal — each schema should be as self-contained as possible.

    2. Liquibase Migrations

    File Structure

    src/main/resources/db/changelog/
├── db.changelog-master.xml          # Master changelog (includeAll)
└── sql/
    ├── 01-profiles-schema.sql       # Profiles DDL + seed data
    ├── 02-shopping-schema.sql       # Shopping DDL
    └── 03-seed-data.sql             # Reference data

    The master changelog uses includeAll to load all SQL files in alphabetical order:

    <includeAll path="db/changelog/sql/" relativeToChangelogFile="false"/>

    Changeset Format

    Use --liquibase formatted sql with --changeset author:id tags.

    --liquibase formatted sql

--changeset jorge:04-add-shopping-tags
CREATE TABLE shopping.tags (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(50) NOT NULL UNIQUE,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    version BIGINT NOT NULL DEFAULT 0
);

    File Naming

    New migration files follow the pattern <NN>-<description>.sql where NN is the next sequential number.

    Changesets Are Immutable

    Once a changeset has been applied to any environment, it must never be modified. Liquibase tracks changesets by checksum — modifying an applied changeset causes a checksum mismatch and breaks deployment. Always add new changesets for schema changes.

    Backward Compatibility

    Migrations must be backward-compatible with the currently running application:

    Safe Migration Rules

    • Add new columns as NULL or with a DEFAULT value.
    • Add new tables freely.
    • Add new indexes freely.
    • NEVER add NOT NULL to an existing column without a default.
    • NEVER rename a column in a single step — use add → migrate → drop across releases.
    • NEVER drop a column that the running application still reads.

    3. Naming Conventions

    All identifiers use snake_case.

    Element Convention Example
    Schema Lowercase domain name profiles, shopping
    Table Plural noun, schema-qualified shopping.shopping_lists
    Column Descriptive snake_case created_by, family_role_id, due_date
    Primary Key id (always) id BIGSERIAL PRIMARY KEY
    Foreign Key fk_<table>_<referenced> fk_list_items_list
    Index idx_<table>_<columns> idx_user_email
    Unique Index idx_<table>_<columns> or inline UNIQUE idx_shopping_items_name_category
    Unique Constraint Inline UNIQUE on column email VARCHAR(100) NOT NULL UNIQUE

    4. Data Types

    Use Case Type Notes
    Primary keys BIGSERIAL or BIGINT GENERATED BY DEFAULT AS IDENTITY Auto-incrementing
    Foreign keys BIGINT Matches referenced PK type
    Short text VARCHAR(n) Length must match backend validation (e.g., email VARCHAR(100))
    Long text TEXT Photos (base64), descriptions, markdown
    Money DECIMAL(10,2) Never use FLOAT for monetary values
    Booleans BOOLEAN NOT NULL DEFAULT <value> Always provide a default
    Timestamps TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP For audit columns
    Dates DATE For date-only values (e.g., birthdate)
    Counters/versions BIGINT NOT NULL DEFAULT 0 or INTEGER DEFAULT 0 For optimistic locking

    VARCHAR Lengths Must Match Backend

    Column lengths must align with Jakarta validation annotations on the corresponding DTO. For example, email VARCHAR(100) matches @Size(max = 100) on UserProfileDTO.email.

    5. Mandatory Audit Columns

    Every table MUST include the following columns:

    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
version BIGINT NOT NULL DEFAULT 0
    Column Purpose
    created_at Immutable record creation timestamp
    updated_at Updated on every modification (managed by Hibernate @UpdateTimestamp)
    version Optimistic locking — prevents lost updates during concurrent edits

    Exception: Append-Only Tables

    History/log tables (e.g., shopping_item_price_history) use recorded_at instead and omit updated_at/version since rows are never modified.

    6. Indexing Strategy

    When to Add Indexes

    • Unique business keys — UNIQUE on natural identifiers (email, name).
    • Foreign key columns — always index FK columns for join performance.
    • Frequent query filters — index columns used in WHERE clauses (e.g., status, used).
    • Composite indexes — for multi-column queries, order columns by selectivity.
    • AVOID indexing low-cardinality boolean columns alone — combine with other columns (e.g., idx_coupons_status_due(used, due_date)).
    -- Unique business key
CREATE INDEX idx_user_email ON profiles.user(email);

-- Composite index for query optimization (most selective first, DESC for recent-first)
CREATE INDEX idx_price_history_item_store
    ON shopping.shopping_item_price_history(item_id, store_id, recorded_at DESC);

-- Combined filter index for dashboard widget
CREATE INDEX idx_coupons_status_due ON shopping.coupons(used, due_date);

    7. Foreign Key Patterns

    Strategy Use When Example
    ON DELETE CASCADE Child has no meaning without parent shopping_list_items → shopping_lists
    ON DELETE SET NULL Reference is optional, child survives shopping_list_items.store_id → shopping_stores
    No action (RESTRICT) Deletion should be blocked shopping_lists.created_by → profiles.user 
    -- CASCADE: items are deleted with their list
CONSTRAINT fk_list_items_list FOREIGN KEY (list_id)
    REFERENCES shopping.shopping_lists(id) ON DELETE CASCADE,

-- SET NULL: item keeps its record but loses store reference
CONSTRAINT fk_list_items_store FOREIGN KEY (store_id)
    REFERENCES shopping.shopping_stores(id) ON DELETE SET NULL

-- RESTRICT (default): cannot delete a user who created lists
CONSTRAINT fk_shopping_lists_user FOREIGN KEY (created_by)
    REFERENCES profiles.user(id)

    8. Seed Data

    Reference data (categories, stores, default roles) is managed in dedicated seed changesets using idempotent inserts:

    INSERT INTO shopping.shopping_categories (name, description, icon) VALUES
('Fruits & Vegetables', 'Fresh produce', 'IconApple'),
('Dairy & Eggs', 'Milk, cheese, yogurt', 'IconEgg')
ON CONFLICT (name) DO NOTHING;

    Seed Data Rules

    • Separate seed data from DDL in its own changeset file.
    • Use ON CONFLICT ... DO NOTHING for idempotency.
    • Only seed reference/lookup data — never seed transactional data.

    9. JPA Alignment

    Hibernate is configured with ddl-auto: validate — it verifies the schema on startup but never modifies it.

    Schema Changes Must Go Through Liquibase

    All DDL changes (tables, columns, indexes, constraints) MUST be added as Liquibase changesets. Never rely on Hibernate to create or alter schema objects. Entity annotations must match the Liquibase-managed schema exactly:

    @Entity
@Table(name = "shopping_lists", schema = "shopping")  // Must match Liquibase DDL
public class ShoppingList {

    AI Alignment

    Directives for AI Agents

    • ALL schema changes MUST be Liquibase changesets — never modify the database directly.
    • NEVER modify an applied changeset — always create new changesets.
    • ALWAYS include created_at, updated_at, and version on new tables.
    • ALWAYS use snake_case for all database identifiers.
    • ALWAYS add indexes on foreign key columns.
    • ALWAYS use BIGSERIAL for primary keys and BIGINT for foreign keys.
    • ALWAYS use DECIMAL(10,2) for monetary values — never FLOAT.
    • ALWAYS use ON CONFLICT ... DO NOTHING for seed data inserts.
    • ALWAYS match VARCHAR(n) lengths with backend Jakarta validation constraints.
    • PREFER ON DELETE CASCADE for parent-child, ON DELETE SET NULL for optional references.
    • ENSURE @Table(name, schema) annotations match the Liquibase-managed schema exactly.
    • ENSURE migrations are backward-compatible — no NOT NULL without defaults on existing columns.
