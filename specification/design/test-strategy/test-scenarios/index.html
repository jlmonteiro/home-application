    Test Scenarios

    Legend

    • Positive Case: Verifies expected behavior under normal conditions.
    • Negative Case: Verifies system resilience and error handling under failure or invalid conditions.

    Authentication & Onboarding

    TS-1: Google OAuth2 User Registration

    • Given: A user who has never logged in to the application.
    • When: The user successfully authenticates via Google OAuth2.
    • Then: A new local user record and profile are created.

    Validates

    • FR-1: Google OAuth2 Authentication
    • FR-2: Automatic User Registration

    TS-2: Google OAuth2 Login (Returning User)

    • Given: A user who already has a local account.
    • When: The user successfully authenticates via Google OAuth2.
    • Then: The existing local user record is retrieved.

    Validates

    • FR-1: Google OAuth2 Authentication

    TS-3: Authentication Failure

    • Given: A user who cancels the login process at Google.
    • When: The user is redirected back to the application.
    • Then: The system redirects them to /login?error=true.

    Validates

    • FR-1: Google OAuth2 Authentication

    TS-4: Service Timeout / Token Exchange Failure

    • Given: An issue with Google's token endpoint.
    • When: The backend attempts to exchange the authorization code for a token.
    • Then: The system displays a "Service Unavailable" error message.

    Validates

    • FR-1: Google OAuth2 Authentication

    User Profile Management

    TS-7: Successful User Profile Update

    • Given: An authenticated user.
    • When: The user sends a PUT /api/user/me with valid social links, a phone number, and a new photo.
    • Then: The system updates the profile, returns 200 OK with the updated HATEOAS resource, and displays a success notification (toast) in the UI.

    Validates

    • FR-4: Update User Profile

    TS-8: Profile Update Validation Failure

    • Given: An authenticated user.
    • When: The user sends a PUT /api/user/me with an invalid phone format or a malformed Facebook URL.
    • Then: The system returns 400 Bad Request with a list of field-level violations.

    Validates

    • FR-4: Update User Profile

    TS-9: Protection of Immutable Fields

    • Given: An authenticated user.
    • When: The user sends a PUT /api/user/me containing a new firstName or email.
    • Then: The system updates the allowed profile fields but ignores the changes to firstName and email.

    Validates

    • FR-4: Update User Profile

    TS-10: Profile Dropdown Detailed View (Conditional)

    • Given: An authenticated user with a mobile phone and a LinkedIn profile.
    • When: The user opens the header dropdown menu.
    • Then: The system displays the mobile phone and a clickable LinkedIn icon/link.

    Validates

    • FR-3: User Profile Quick View

    TS-11: Profile Dropdown Compact View (Fallback)

    • Given: An authenticated user with no optional profile fields (phone, social).
    • When: The user opens the header dropdown menu.
    • Then: The system displays only basic info (name, photo) and the "View/Edit Profile" link, omitting phone and social sections.

    Validates

    • FR-3: User Profile Quick View

    Shopping Module Scenarios

    TS-23: Category & Master Item Creation

    • Given: An authenticated user.
    • When: The user creates a "Dairy" category and then adds a "Whole Milk" item linked to it.
    • Then: The system SHALL persist the category and the item, ensuring the link is established.

    Validates

    • FR-5: Category & Item Management

    TS-30: Store Management with Photos

    • Given: An authenticated user.
    • When: The user creates a new Store with a name, description, and an uploaded photo.
    • Then: The system SHALL save the store details and correctly display the photo in the list.

    Validates

    • FR-6: Store Management

    TS-31: Shopping List Visibility & Collaboration

    • Given: User A creates a "Weekend Party" shopping list.
    • When: User B logs in.
    • Then: User B SHALL see the "Weekend Party" list on their dashboard and be able to add items to it.

    Validates

    • FR-7: Shopping List Creation & Collaboration

    TS-13: Collaborative Item Check-off (Real-time)

    • Given: Two users (A and B) belong to the same household and are viewing the same shopping list.
    • When: User A marks "Apples" as "Bought".
    • Then: The backend updates the item status, and User B's UI automatically reflects the "Bought" state.

    Validates

    • FR-7: Shopping List Creation & Collaboration
    • FR-10: In-Store Progress Tracking

    TS-14: Intelligent Price Suggestions

    • Given: A user has previously bought "Bread" at "Store X" for €1.50 and "Store Y" for €1.20.
    • When: The user adds "Bread" to a new list and selects "Store X".
    • Then: The system suggests €1.50 as the price.

    Validates

    • FR-9: Price History & Suggestions

    TS-25: Global Price Suggestion Fallback

    • Given: A user has previously bought "Bread" at "Store X" for €1.50 (most recent) and "Store Y" for €1.20.
    • When: The user adds "Bread" to a list and selects "Any Place".
    • Then: The system SHALL suggest €1.50 as the Global Last Price.

    Validates

    • FR-9: Price History & Suggestions

    TS-15: Automatic Data Retention (Cleanup)

    • Given: A shopping list that was marked as "Completed" 95 days ago.
    • When: The daily retention background task executes.
    • Then: The system SHALL permanently delete the list and all its associated ListItem records.

    Validates

    • FR-11: Automatic Data Retention

    TS-17: Loyalty Card Barcode Validation

    • Given: A loyalty card configured with the "Code 128" barcode type.
    • When: The user views the card details in the UI.
    • Then: The system SHALL render a linear 1D barcode encoding the exact card number.

    Validates

    • FR-12: Loyalty Cards

    TS-24: Coupon Usage Tracking & Filtering

    • Given: An authenticated user with an active coupon.
    • When: The user marks the coupon as "Used".
    • Then: The coupon SHALL be hidden from the default active view and the Dashboard warning panel.

    Validates

    • FR-13: Store Coupons

    TS-16: Dashboard Coupon Warning

    • Given: An unused coupon for "Lidl" expiring in 48 hours.
    • When: The user views the application Dashboard.
    • Then: The "Expiring Coupons" panel displays the Lidl coupon with its expiration date highlighted.

    Validates

    • FR-15: Expiration Warning Panel (Dashboard)

    Family Hierarchy & Age Groups

    TS-18: First User Bootstrap (Onboarding)

    • Given: A completely empty system with no users.
    • When: The first user successfully authenticates via Google.
    • Then: The system SHALL assign them the "Adult" age group, regardless of their actual age.

    Validates

    • UJ-1: First-Time Login (Onboarding)

    TS-19: Automated Age Group Classification

    • Given: Configured age ranges: Child (0-12), Teenager (13-17), Adult (18+).
    • And: A user with a birthdate exactly 13 years ago today.
    • When: The user logs in or the daily recalculation task runs.
    • Then: The user's profile SHALL be updated from "Child" to "Teenager".

    Validates

    • FR-16: Automated Age Group Classification

    TS-20: Google Birthdate Synchronization

    • Given: An authenticated session with the user.birthday.read scope.
    • When: The onboarding process calls the Google People API.
    • Then: The system SHALL extract the birthdate and persist it in the UserProfile.
    • And: If Google returns no birthdate, the system SHALL prompt the user for manual entry.

    Validates

    • FR-17: Google Birthdate Integration

    TS-28: Age Range Update & Global Recalculation

    • Given: An "Adult" user.
    • When: The user modifies the "Teenager" range in the Settings module.
    • Then: The system SHALL trigger a recalculation of age groups for all household members based on the new ranges.

    Validates

    • FR-18: Age Group Configuration
    • UJ-5: Configuring Household Age Groups

    TS-27: Age Range Contiguity & Overlap Validation

    • Given: An "Adult" user.
    • When: The user attempts to save age ranges that overlap (e.g., Child 0-13, Teenager 12-18).
    • Then: The system SHALL return a validation error and prevent saving.

    Validates

    • FR-18: Age Group Configuration

    TS-26: Custom Family Role Management

    • Given: An "Adult" user.
    • When: The user creates a custom role "Grandmother" in the Settings module.
    • Then: The role SHALL become available for selection in user profiles.

    Validates

    • FR-19: Family Role Configuration

    TS-21: Settings Access Control

    • Given: A user classified as "Child" or "Teenager".
    • When: They attempt to access GET /api/settings/** or PUT /api/settings/**.
    • Then: The system SHALL return a 403 Forbidden status.

    Validates

    • FR-18: Age Group Configuration
    • FR-19: Family Role Configuration

    Infrastructure & Reliability

    TS-5: Database Offline

    • Given: The PostgreSQL database is unreachable.
    • When: An API request is made to /api/user/me.
    • Then: The system returns a 503 Service Unavailable with an RFC 7807 Problem Detail.

    Validates

    • NFR-3: Reliability & Sync

    TS-6: Unauthorized Access (Zero Trust)

    • Given: An unauthenticated request.
    • When: Accessing /api/user/me.
    • Then: The system returns a 401 Unauthorized status.

    Validates

    • NFR-1: Security (Zero Trust)

    TS-12: Frontend Cache Utilization

    • Given: An authenticated user who has already loaded their profile.
    • When: The user navigates between the Dashboard and Profile pages within a 5-minute window.
    • Then: The system SHALL NOT trigger a new network request to /api/user/me, serving the data from the TanStack Query cache instead.

    Validates

    • NFR-2: Performance (Latency)

    TS-22: Documentation Generation Verification

    • Given: A valid mkdocs.yml and markdown files in docs/.
    • When: The Gradle task ./gradlew generateDocs is executed.
    • Then: A directory build/docs/ SHALL be created containing a functional static HTML site.

    Validates

    • FR-20: Integrated Documentation Build (MkDocs)
    • FR-21: Gradle Integration

    Recipes Module Scenarios

    TS-30B: Recipe CRUD with Photos

    • Given: An authenticated user.
    • When: The user creates a recipe with a name, markdown description, two photos, and sets one as default.
    • Then: The system SHALL persist the recipe and photos, and the recipes list SHALL display the default photo.

    Validates

    • FR-23: Recipe Management
    • FR-24: Recipe Photos

    TS-31B: Dynamic Label Lifecycle

    • Given: An authenticated user creating a recipe.
    • When: The user types "Italian" (new label) and "Comfort Food" (existing label).
    • Then: The system SHALL create "Italian" in the labels table and link both labels to the recipe.
    • When: The user later removes "Italian" from the only recipe that uses it.
    • Then: The system SHALL delete "Italian" from the labels table.

    Validates

    • FR-23: Recipe Management

    TS-32: Ingredient Linking with Unit Validation

    • Given: A recipe and a shopping item "Chicken Breast" in the catalog.
    • When: The user adds "Chicken Breast" as an ingredient with quantity 500 and unit G.
    • Then: The system SHALL persist the ingredient with the same unit enum used by the Shopping module.

    Validates

    • FR-25: Recipe Ingredients

    TS-33: Step Reordering

    • Given: A recipe with 3 preparation steps in order [A, B, C].
    • When: The user drags step C to position 1.
    • Then: The system SHALL update sort_order values so the order becomes [C, A, B] in a single transaction.

    Validates

    • FR-27: Preparation Steps

    TS-34: On-the-Fly Nutrition Calculation

    • Given: A recipe with 2 ingredients: "Rice" (200g, nutrition entries: [calories: 130 kcal, protein: 2.7 g]) and "Chicken" (300g, nutrition entries: [calories: 165 kcal, protein: 31 g]).
    • When: The user views the recipe detail.
    • Then: The system SHALL display aggregated totals grouped by nutrient key: [calories: 295 kcal, protein: 14.7 g], computed on-the-fly.

    Validates

    • FR-26: Nutrition Data

    TS-35: Nutrition Calculation with Missing Data

    • Given: A recipe with 2 ingredients, one with nutrition entries and one without.
    • When: The user views the recipe detail.
    • Then: The system SHALL calculate totals from the ingredient with entries and indicate which ingredient is missing nutrition data.

    Validates

    • FR-26: Nutrition Data

    TS-36: Rating Average with Individual Votes

    • Given: User A rates a recipe 4 stars, User B rates it 5 stars.
    • When: Any user views the recipe.
    • Then: The system SHALL display an average of 4.5 and, when expanded, show User A (4 stars) and User B (5 stars) with timestamps.

    Validates

    • FR-29: Recipe Ratings

    TS-37: Rating Uniqueness

    • Given: User A has already rated a recipe 3 stars.
    • When: User A submits a new rating of 5 stars.
    • Then: The system SHALL update the existing rating to 5 (not create a duplicate).

    Validates

    • FR-29: Recipe Ratings

    TS-38: Recipe Comment Notification

    • Given: User A created a recipe.
    • When: User B adds a comment to the recipe.
    • Then: The system SHALL create a NEW_RECIPE_COMMENT notification for User A.

    Validates

    • FR-28: Recipe Comments

    TS-39: Recipe Deletion RESTRICT

    • Given: A recipe referenced in an active meal plan.
    • When: A user attempts to delete the recipe.
    • Then: The system SHALL return a validation error and prevent deletion.

    Validates

    • FR-23: Recipe Management

    Meals Module Scenarios

    TS-40: Meal Time Configuration

    • Given: An authenticated user.
    • When: The user creates "Lunch" with time 12:00 on weekdays and 13:00 on weekends.
    • Then: The system SHALL persist the meal time with 7 schedule entries.

    Validates

    • FR-30: Meal Time Configuration

    TS-41: Meal Plan Creation (Week Boundary)

    • Given: An authenticated user.
    • When: The user creates a meal plan for the week of Monday April 13, 2026.
    • Then: The system SHALL create a plan with week_start_date = 2026-04-13.

    Validates

    • FR-31: Weekly Meal Plan

    TS-42: Duplicate Week Plan

    • Given: A meal plan already exists for week of April 13, 2026.
    • When: A user attempts to create another plan for the same week.
    • Then: The system SHALL return a validation error due to the unique constraint on week_start_date.

    Validates

    • FR-31: Weekly Meal Plan

    TS-43: Multi-Recipe Meal Assignment

    • Given: A meal plan with a Monday dinner entry.
    • When: The user assigns "Grilled Chicken" and "Caesar Salad" to the entry for the entire household.
    • Then: The system SHALL create two meal_plan_entry_recipes records with null user_id.

    Validates

    • FR-31: Weekly Meal Plan

    TS-44: Per-Member Assignment

    • Given: A meal plan with a Monday breakfast entry.
    • When: The user assigns "Scrambled Eggs" to Jorge and "Toast" to Fernanda.
    • Then: The system SHALL create separate meal_plan_entry_recipes records with the respective user_id values.

    Validates

    • FR-31: Weekly Meal Plan

    TS-45: Meal Plan Publish & Notify

    • Given: A PENDING meal plan with entries.
    • When: The creator clicks "Notify Household".
    • Then: The system SHALL set status to PUBLISHED and create MEAL_PLAN_PUBLISHED notifications for all household members.

    Validates

    • FR-32: Meal Plan Publish & Notify

    TS-48: Thumbs Up/Down on Meal

    • Given: A meal entry marked as "done".
    • When: User A gives a thumbs-up.
    • Then: The system SHALL create a record in meal_plan_votes with vote = true for User A on that entry.
    • When: User A changes their vote to thumbs-down.
    • Then: The system SHALL update the existing vote record to vote = false.

    Validates

    • FR-37: Meal Thumbs Up/Down

    TS-50: Meal Plan Retention

    • Given: A meal plan with week_start_date 11 weeks ago.
    • When: The daily retention task executes at 03:00 AM.
    • Then: The system SHALL permanently delete the plan and all associated entries, recipes, and member records.

    Validates

    • FR-34: Meal Plan Data Retention

    TS-51: Shopping List Export Preview

    • Given: A meal plan with recipes requiring "Eggs" (6 UNIT) and "Milk" (1 L). An existing shopping list already has "Eggs" (3 UNIT).
    • When: The user requests an export preview targeting the existing list.
    • Then: The system SHALL return: existing items = [Eggs: 3→9 UNIT], new items = [Milk: 1 L].

    Validates

    • FR-36: Shopping List Integration

    TS-52: Shopping List Export Confirm

    • Given: A confirmed export preview.
    • When: The user confirms the export.
    • Then: The system SHALL increase "Eggs" quantity to 9 and add "Milk" (1 L) with no store assigned.

    Validates

    • FR-36: Shopping List Integration

    TS-53: Mark Meal as Done

    • Given: A meal plan entry that is not done.
    • When: A user marks it as done.
    • Then: The system SHALL set is_done = true.

    Validates

    • FR-31: Weekly Meal Plan

    Notifications & Messaging Scenarios

    TS-54: Notification Delivery & Unread Count

    • Given: User A has 3 unread notifications.
    • When: User A queries the unread count endpoint.
    • Then: The system SHALL return { count: 3 }.

    Validates

    • FR-38: In-App Notifications

    TS-55: Mark Notification as Read

    • Given: An unread notification for User A.
    • When: User A marks it as read.
    • Then: The system SHALL set read = true and the unread count SHALL decrease by 1.

    Validates

    • FR-38: In-App Notifications

    TS-56: Send and Receive Message

    • Given: User A and User B in the same household.
    • When: User A sends a message to User B.
    • Then: The system SHALL persist the message, create a NEW_MESSAGE notification for User B, and the message SHALL appear in both users' conversation view.

    Validates

    • FR-39: User Messaging

    TS-57: Message Read Status

    • Given: User B has an unread message from User A.
    • When: User B opens the conversation with User A.
    • Then: The system SHALL mark the message as read.

    Validates

    • FR-39: User Messaging

    Summary & Environment

    • Testcontainers: Used for PostgreSQL 17 integration.
    • Mocks: Google OAuth2 and People API are mocked for deterministic testing.
    • Verification: 100% of BDD scenarios (Happy & Sad) MUST pass.
    • Integrity: RFC 7807 validation for all error responses.
    • Immutability: email, firstName, and lastName remain unchanged after any profile update.

    New Scenarios (Post-Implementation)

    TS-58: Centralized Photo Upload & Retrieval

    • Given: An authenticated user.
    • When: The user uploads a photo via any module (profile, recipe, item, store).
    • Then: The system SHALL store the binary data in media.photos and return a URL reference (/api/images/{name}).
    • When: The URL is requested via GET.
    • Then: The system SHALL return the binary image with the correct Content-Type header.

    Validates

    • FR-42: Centralized Media Service

    TS-59: Standalone Item Assignment to Meal Entry

    • Given: A meal plan with a Monday breakfast entry.
    • When: The user assigns "Fresh Orange Juice" (2 UNIT) directly to the entry for the entire household.
    • Then: The system SHALL create a meal_plan_entry_items record with item_id, quantity=2, unit=UNIT, and user_id=null.

    Validates

    • FR-41: Standalone Meal Items

    TS-60: Nutrient Master Catalog CRUD

    • Given: An authenticated Adult user.
    • When: The user creates a new nutrient "Omega-3" with unit "mg".
    • Then: The system SHALL persist the nutrient in recipes.nutrients.
    • When: The user attempts to delete a nutrient referenced by nutrition entries.
    • Then: The system SHALL return a validation error and prevent deletion.

    Validates

    • FR-40: Nutrient Master Catalog

    TS-61: Ingredient Grouping

    • Given: A recipe with ingredients.
    • When: The user adds "Flour" with group_name = "For the dough" and "Tomato Sauce" with group_name = "For the topping".
    • Then: The system SHALL persist the group_name and the UI SHALL display ingredients grouped by their group name.

    Validates

    • FR-25: Recipe Ingredients

    TS-62: Coupon Barcode Rendering

    • Given: A coupon with code = "SAVE10" and barcode_type = "QR".
    • When: The user views the coupon details.
    • Then: The system SHALL render a QR code encoding "SAVE10".

    Validates

    • FR-13: Store Coupons

    TS-63: Shopping List Item Unavailability

    • Given: A shopping list with an item "Organic Milk".
    • When: The user marks "Organic Milk" as unavailable.
    • Then: The system SHALL set unavailable = true and visually distinguish it from bought and pending items.

    Validates

    • FR-10: In-Store Progress Tracking

    TS-64: Recipe Multiplier in Meal Plan

    • Given: A meal plan entry with "Pancakes" assigned.
    • When: The user sets the multiplier to 2.0.
    • Then: The system SHALL persist multiplier = 2.0 on the meal_plan_entry_recipes record and use it for shopping list export quantity calculations.

    Validates

    • FR-31: Weekly Meal Plan

    TS-65: Shopping Item Enriched Fields

    • Given: An authenticated user creating a shopping item "Milk (1L Bottle)".
    • When: The user sets unit = "pcs", pc_quantity = 1.0, pc_unit = "L", nutrition_sample_size = 100, nutrition_sample_unit = "ml".
    • Then: The system SHALL persist all fields and use them for unit conversion and nutrition calculations.

    Validates

    • FR-5: Category & Item Management

    TS-66: Meal Plan Export with Standalone Items

    • Given: A meal plan with a recipe requiring "Eggs" (6 UNIT) and a standalone item "Orange Juice" (2 L).
    • When: The user requests an export preview.
    • Then: The system SHALL include both recipe ingredients and standalone items in the merge preview.

    Validates

    • FR-36: Shopping List Integration
    • FR-41: Standalone Meal Items
